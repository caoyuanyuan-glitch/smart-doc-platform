# Requirements Document

## Introduction

本文档定义审核模块重构需求。目标是将当前耦合在单一流程中的规则检查、AI 审核、误报过滤、结果展示和统计拆分为可验证、可观测、可回归测试的审核引擎，提高审核准确性、运行效率和同一文档多次审核的一致性。

## Glossary

- **审核引擎**: 执行文档解析、候选问题发现、问题验证、去重聚合、结果保存的后端能力。
- **确定性规则**: 基于正则、结构解析、术语库或固定逻辑执行的规则，输出应具备可重复性。
- **AI 候选问题**: LLM 发现的候选问题，必须通过后端证据校验后进入正式结果。
- **正式问题**: 通过证据、建议和语义保护校验后保存到数据库的问题。
- **展示问题**: 报告中合并展示的同类问题，不改变底层正式问题数量。
- **评测集**: 带标准答案、误报样本和漏检样本的固定文档集合。
- **敏感实体**: 数字值、单位、产品型号、试剂名、供应方或用户角色、合规声明、存储动作、技术术语、页码和日期。

## Requirements

### Requirement 1: 分层审核流水线

**User Story:** AS 文档审核用户, I want 审核结果由稳定流水线生成, so that 同一文档多次审核结果保持一致。

#### Acceptance Criteria

1. WHEN 用户创建审核任务, 审核引擎 SHALL 按照“解析上下文、确定性规则、AI 候选发现、问题验证、去重聚合、结果保存”的顺序执行。
2. WHEN 任一阶段完成, 审核引擎 SHALL 记录阶段输入数量、输出数量、耗时和过滤数量。
3. IF 任一非关键阶段失败, 审核引擎 SHALL 保留已完成阶段的问题并记录失败阶段和错误原因。
4. WHILE 审核任务运行中, 审核引擎 SHALL 提供可轮询的阶段进度和当前阶段说明。

### Requirement 2: 确定性规则优先

**User Story:** AS 文档审核用户, I want 高频客观问题由规则稳定检出, so that 审核结果减少 LLM 波动。

#### Acceptance Criteria

1. WHEN 文档包含单位、日期、URL、产品型号、图表编号、版权年份或修订历史问题, 审核引擎 SHALL 优先使用确定性规则生成候选问题。
2. WHEN 确定性规则生成候选问题, 审核引擎 SHALL 为问题附加规则 ID、证据位置、建议模板和审核依据。
3. IF 确定性规则命中已知例外条件, 审核引擎 SHALL 跳过该候选问题并记录例外原因。
4. WHEN 新增高频问题类型, 审核引擎 SHALL 支持以结构化规则和测试样例加入回归验证。

### Requirement 3: AI 输出受控

**User Story:** AS 文档审核用户, I want AI 只补充可证明的问题, so that AI 不输出风格润色、语义改写和空建议。

#### Acceptance Criteria

1. WHEN AI 生成候选问题, AI 候选问题 SHALL 包含原文片段、建议、问题类型、规则依据、证据理由和置信度。
2. IF AI 候选问题的原文片段无法在文档正文中定位, 审核引擎 SHALL 过滤该候选问题。
3. IF AI 候选问题的建议改变敏感实体, 审核引擎 SHALL 过滤该候选问题。
4. IF AI 候选问题的建议与原文无实质差异, 审核引擎 SHALL 过滤该候选问题。
5. IF AI 候选问题仅涉及可读性、语气或风格润色, 审核引擎 SHALL 过滤该候选问题。

### Requirement 4: 问题生命周期清晰

**User Story:** AS 文档审核管理员, I want 底层问题、展示问题和统计问题分开管理, so that 统计和报告保持一致。

#### Acceptance Criteria

1. WHEN 候选问题通过校验, 审核引擎 SHALL 保存正式问题并保留来源阶段。
2. WHEN 报告展示同类问题, 报告生成器 SHALL 合并展示问题并保留底层正式问题引用。
3. WHEN 用户标记误报, 审核引擎 SHALL 记录人工判断并应用于同文档后续展示和统计。
4. WHEN 生成看板统计, 统计服务 SHALL 使用正式问题和人工判断计算检出率。

### Requirement 5: 评测闭环

**User Story:** AS 产品负责人, I want 每次审核逻辑变更都有可量化评测, so that 准确性改进可以被证明。

#### Acceptance Criteria

1. WHEN 审核引擎代码变更, 评测工具 SHALL 支持运行固定评测集并输出检出率、误报率、一致性和耗时。
2. WHEN 评测集包含标准答案, 评测工具 SHALL 区分平台检出、人工补录、明确误报和允许忽略项。
3. WHEN 同一文档连续审核多次, 评测工具 SHALL 输出正式问题集合的一致性比例。
4. IF 评测指标低于基线阈值, 评测工具 SHALL 返回失败状态并输出退化项。

### Requirement 6: 运行效率和资源控制

**User Story:** AS 系统管理员, I want 长文档审核运行时间可控, so that 用户获得稳定响应。

#### Acceptance Criteria

1. WHEN 文档长度超过 AI 单次处理阈值, 审核引擎 SHALL 使用分块审核并限制每块最大字符数。
2. WHEN AI 阶段超过配置的超时时间, 审核引擎 SHALL 保存规则阶段结果并标记 AI 阶段超时。
3. WHEN 多个 provider 可用, 审核引擎 SHALL 按配置优先级调用 provider 并记录 provider 名称和耗时。
4. IF provider 返回空结果或格式错误, 审核引擎 SHALL 尝试下一个可用 provider 并记录失败原因。

### Requirement 7: 可观测性和排障

**User Story:** AS 开发人员, I want 审核任务有结构化诊断信息, so that 低效和不稳定问题可以快速定位。

#### Acceptance Criteria

1. WHEN 审核任务完成, 审核引擎 SHALL 保存阶段级诊断摘要。
2. WHEN 候选问题被过滤, 审核引擎 SHALL 记录过滤规则和过滤原因。
3. WHEN 用户查看审核详情, 系统 SHALL 支持查看问题来源、验证状态和人工判断状态。
4. WHEN 审核结果数量异常波动, 系统 SHALL 提供与历史运行的阶段数量对比。
